## Challenges and How They Were Addressed

### Challenge 1: Coordinating Multi-Agent Workflow

**Problem:** Five agents needed to run in strict dependency order (Research Verifier after Researcher, Bug Fixer after Planner, Security Verifier and Unit Test Generator in parallel after Fixer), with no manual steps between stages.

**Solution:** Designed a `/pipeline` skill in `.claude/skills/pipeline/SKILL.md` that:
- Reads the bug-context.md to determine which bug to work on
- Orchestrates each stage, passing output artifacts from one stage to the next
- Dispatches Security Verifier and Unit Test Generator in parallel using the `--run_in_background` parameter
- Waits for both to complete before reporting final results
- Explicitly assigns models to each agent based on task complexity

### Challenge 2: Research Quality Measurement

**Problem:** How do we know if the Bug Researcher's output is accurate? Without a systematic way to rate quality, the Research Verifier agent had no framework.

**Solution:** Created the `research-quality-measurement` skill defining four quality levels:
- **Exemplary:** All file:line references verified, complete snippet matches, no discrepancies
- **Good:** All references verified, minor formatting discrepancies only
- **Acceptable:** >90% references verified, some snippets have partial matches
- **Poor:** <90% verified, significant discrepancies or missing context

The Research Verifier uses this skill to justify its quality rating in the output.

### Challenge 3: Security Review Rigor

**Problem:** The security verifier needed to find real issues (not just obvious ones) but also distinguish between production-critical findings and demo-app concerns.

**Solution:** Used `claude-opus-4-8` (reasoning-focused model) for the security review and structured the scan to check for injection, hardcoded secrets, insecure comparison, missing validation, and unsafe dependencies. The agent rated each finding by severity (CRITICAL/HIGH/MEDIUM/LOW/INFO) and provided specific remediation steps. The MEDIUM finding about test-committed secrets is a real issue that would fail secret scanners.

### Challenge 4: FIRST-Compliant Test Generation

**Problem:** Generated tests needed to follow best practices (Fast, Independent, Repeatable, Self-validating, Timely), not just "pass."

**Solution:** Created the `unit-tests-first` skill defining the FIRST framework. The Unit Test Generator agent uses this skill and verifies:
- Each test runs in <100ms (Fast)
- Each test sets up its own fixtures and tears down in finally (Independent)
- Tests pass consistently across runs (Repeatable)
- Tests assert on concrete values, not side effects (Self-validating)
- Tests are written alongside the code being changed (Timely)

### Challenge 5: Parallel Execution Without Race Conditions

**Problem:** Security Verifier and Unit Test Generator both read the same changed source files and could interfere if run naively in parallel.

**Solution:** Both agents read files without modifying them; they only produce report artifacts to separate locations (`security-report.md` vs. `test-report.md`). The pipeline dispatches both with `--run_in_background` and the top-level skill waits for both to complete before declaring the pipeline done.
